## Project Overview

KiTTY is a Windows-only fork of PuTTY 0.76 that adds session filtering, portability (INI file storage), automatic password/command, session launcher, URL hyperlinks, background images, transparency, ZModem, a hidden text editor, and more. Written in C, cross-compiled from Linux via MinGW to produce Windows executables.

## Build Commands

All builds run from `0.76b_My_PuTTY/windows/`:

```bash
# Build main executable (local MinGW)
make -f MAKEFILE.MINGW putty.exe

# Full KiTTY build (produces all tools)
make -f MAKEFILE.MINGW 9bis

# Cross-compile 32-bit (requires i686-w64-mingw32-gcc)
make -f MAKEFILE.MINGW cross

# Cross-compile 64-bit (requires x86_64-w64-mingw32-gcc)
make -f MAKEFILE.MINGW cross64

# Clean build artifacts
make -f MAKEFILE.MINGW clean

# Build with explicit tool path (e.g., 64-bit cross)
make -e TOOLPATH=/usr/bin/x86_64-w64-mingw32- -f MAKEFILE.MINGW putty.exe
```

The CI uses the `cyd01/cross-gcc` Docker image. The dev container in `.devcontainer/` replicates this environment.

## Architecture

### Fork-and-Patch Structure

The repo contains two PuTTY source trees:
- `0.76_My_PuTTY/` — original PuTTY 0.76 (reference only, not built)
- `0.76b_My_PuTTY/` — modified PuTTY 0.76b (active build source)

KiTTY-specific code lives at the repository root (`kitty.c`, `kitty_*.c/h`) and is `#include`-d or linked into the PuTTY build. The PuTTY core is modified in-place with `#ifdef MOD_*` guards.

### Feature Flag System (MOD_*)

All KiTTY additions are gated behind `MOD_*` preprocessor defines set in `MAKEFILE.MINGW`. Active flags:

`MOD_PERSO` (master flag), `MOD_WTS`, `MOD_STARTBUTTON`, `MOD_SAVEDUMP`, `MOD_DISABLEALTGR`, `MOD_KEYMAPPING`, `MOD_PUTTYX`, `MOD_PORTKNOCKING`, `MOD_HYPERLINK`, `MOD_PRINTCLIP`, `MOD_LAUNCHER`, `MOD_RECONNECT`, `MOD_RUTTY`, `MOD_TUTTY`, `MOD_ADB`, `MOD_BACKGROUNDIMAGE`, `MOD_ZMODEM`, `MOD_PROXY`, `MOD_FAR2L`

Optional flags (commented out in Makefile): `MOD_NOPASSWORD`, `MOD_NOTRANSPARENCY`, `MOD_NOHYPERLINK`, `MOD_PORTABLE`, `MOD_TUTTYCOLOR`, `MOD_INTEGRATED_AGENT`, `MOD_INTEGRATED_KEYGEN`

### Key Source Files

- `kitty.c` (5769 lines) — Main KiTTY entry point; includes `kitty_savedump.c` and `kitty_launcher.c` via `#include`; heavy use of `#ifdef MOD_*`
- `kitty_puttytray.c` — File-based session storage (portable mode, replaces registry)
- `kitty_settings.c` — KiTTY-specific configuration UI additions
- `kitty_image.c` — Background image rendering
- `kitty_ini.h` — Default INI configuration compiled into the binary as a string literal
- `kitty_help.h` — Command-line help text
- `0.76b_My_PuTTY/windows/window.c` — Main Windows window procedure (heavily modified from PuTTY)
- `0.76b_My_PuTTY/terminal/terminal.c` — Terminal emulator engine
- `0.76b_My_PuTTY/putty.h` — Master header with KiTTY additions under `MOD_PERSO`

### Prebuilt Static Libraries

These auxiliary libraries ship as prebuilt `.a` archives (both 32-bit and 64-bit variants) and are linked directly rather than compiled from source:

`base64`, `bcrypt`, `blocnote` (hidden text editor), `jpeg`, `md5`, `mini` (INI parser), `regex`

Source is included for some (base64, mini, blocnote) but the build links the archives.

### Portable vs Registry Storage

KiTTY supports two session storage modes: Windows Registry (like PuTTY) and INI-file-based portability. `MOD_PORTABLE` forces portable mode. `kitty_puttytray.c` implements the file-based storage layer.

## Testing

There is no formal test framework or test runner. The only test target in the Makefile is `testcrypt.exe` (cryptographic tests). Testing is manual/ad-hoc — there is no CI test step.

## Important Conventions

- The Makefile warns: "DO NOT EDIT THIS FILE DIRECTLY; edit Recipe or mkfiles.pl instead" — however, KiTTY's `MOD_*` additions are directly in `MAKEFILE.MINGW`, so practical edits go there.
- Comments in KiTTY-specific code are often in French.
- `masterpassword.txt` contains the build-time master password seed.
- The `.ccls` file references `0.73_My_PuTTY/` (outdated) and may need updating for language server accuracy.
- UPX compression is applied to final executables in the `cross` and `cross64` targets.
